# Visual Recap

`/visual-recap` is the reverse of `/visual-plan`. Instead of describing what you
are about to build, it describes what was just built — at a higher altitude than
line-by-line review. A reviewer scans the shape of the change before spending
attention on the literal lines.

The deliverable is a self-contained HTML file written to disk using the same
dual-theme shell as `visual-plan`. No external dependencies.

## When To Use

Build a recap when:
- A PR or commit is large, multi-file, or touches schema, API contracts, or architecture
- A reviewer would benefit from seeing the change mapped to structure before reading the raw diff
- You've just finished a work thread and want to produce a handoff artifact

Skip for small, single-file, or obvious diffs — a recap is review overhead, and a tiny change reviews faster as plain diff.

## Scope — The Whole Work Unit

When invoked after a conversation thread, the default scope is the **whole work unit**, not just the most recent commit. Gather all thread-owned changes: original implementation, bug fixes, UI follow-ups, tests, config changes. Exclude unrelated pre-existing edits. If scope is ambiguous, state the assumption before writing.

When updating a recap after feedback, revise it so it still covers the whole work unit plus the correction — do not replace a broad recap with a narrow recap of only the latest change unless explicitly asked.

## Keep It Lean But Substantial

**Lean:** No boilerplate intro, no disclaimers, no "this recap was generated by…" prose. The file name, title, and file map already carry provenance.

**Substantial:** A recap is not a single diagram plus one sentence. Alongside the visual/structural headline it must carry implementation evidence:
- A **file map** of every changed file with its change type
- **Key diffs** — the 3–8 most load-bearing files, each with a one-line summary and annotations on the important hunks
- **Schema / API blocks** when contracts changed
- **Wireframes** when UI changed (non-negotiable — see below)

## Canonical Shape

Write tabs in this order, top to bottom. Only include tabs with real content:

| Tab | When to include |
|---|---|
| `overview` | Always — 1–3 paragraph outcome narrative: what changed and why |
| `wireframes` | Any diff that changes rendered UI, layout, interaction, or visual state |
| `schema` | Schema migrations, data-model changes, new/removed tables or fields |
| `api` | New, changed, or removed routes, actions, or contracts |
| `files` | File map of every changed file with change type and short note |
| `changes` | Key diffs — 3–8 files, each with summary and annotations |

**Budgets:**
- 3–8 files in the `changes` tab. Fewer than 3 on a large change under-serves the reviewer; more than 8 stops being a summary
- Each diff excerpt: prefer under ~150 lines; summarize or omit the rest of long files
- Title: ≤70 characters. Overview: 1–3 paragraphs

## Before/After Is The Headline

The recap's centre of gravity is comparison:

- **UI changes** → wireframes first, before/after pair when comparison clarifies the change; after-only when purely additive. Show the changed entry point, the opened surface, and the resulting state — not just the first visible affordance.
- **Schema/API changes** → before/after presented as two-column cards (old shape left, new shape right) in the `schema` or `api` tab.
- **Code changes** → split diff (before left, after right) with a one-line summary and a few annotations on the load-bearing hunks. Never leave a diff unlabelled.

## UI Changes Require Wireframes

When the diff changes rendered UI, layout, density, interaction, navigation, or visual state, the recap **must** include wireframes. Prose and file diffs are not a substitute for showing what changed visually.

Before choosing wireframes, make a coverage pass from the diff:
- Entry surface where the change appears (page header, list row, toolbar, route shell)
- Interaction surface that opens (popover, dialog, tab, sheet, dropdown)
- Resulting/destination state (public page, empty state, error state, permission-denied)
- Access or role variants when permissions changed

Rules:
- Match real app chrome — existing sidebars, toolbars, density
- Use `--wf-*` CSS variables from the plan shell (no raw hex inside wireframe blocks)
- Real product content, not Lorem Ipsum
- For tiny surfaces (menus, popovers, toasts), show the focused sub-surface — don't redraw the full page unless placement is part of the change

## Grounding Rule

Structured blocks — file maps, schema changes, API changes, diffs — **must be derived from the actual changed lines**. Real paths, real fields, real method/path, real before/after text. Never inferred, rounded, or invented. The model writes only the prose: the "why", the narrative, the risk read.

When the diff does not contain a fact, leave it out rather than guess. Mark anything inferred (not extracted) as inferred in a note.

A confidently wrong recap is dangerous: a reviewer who trusts the summary may skip the exact line the summary got wrong.

## Security

- **Redact secrets.** A diff can contain API keys, tokens, webhook URLs, signing secrets, or `.env` values. Never copy these into a diff block, file map snippet, or prose. Redact them (`sk-•••`, `<redacted>`).
- **Never transcribe credentials.** Any credential-looking literal in the diff gets redacted, even if it looks like a placeholder.

## Core Workflow

1. **Gather the diff.** Run `git diff`, `git log -p`, or `gh pr diff` to get the actual changed lines. For a PR: `gh pr diff <number>`. For a branch: `git diff main...HEAD`. For the working tree: `git diff HEAD`.
2. **Inventory the scope.** List changed files, changed routes/actions, schema migrations, UI surfaces. Map each to a tab.
3. **UI coverage pass.** If any file in the diff touches components, templates, styles, or layouts — plan wireframes before writing.
4. **Write the HTML file** to the repo root or a `recaps/` directory: `<repo-root>/<branch-or-pr-slug>_recap.html`
5. **Open it.** Run `open <path>` so the reviewer can see it immediately.
6. **Update on feedback.** Re-read the file, apply changes, re-open.

## HTML Shell

Use the full dual-theme shell from `visual-plan` (dark default, light toggle). Add these tab-specific patterns:

```html
<!-- File map entry -->
<div class="file-entry">
  <span class="file-badge added">added</span>   <!-- or: modified, removed, renamed -->
  <code>path/to/file.ts</code>
  <span class="file-note">One-line note about what this file does</span>
</div>

<!-- Split diff block -->
<div class="diff-block">
  <div class="diff-header">
    <code class="diff-filename">path/to/file.ts</code>
    <span class="diff-summary">One-line summary of what this hunk changes</span>
  </div>
  <div class="diff-cols">
    <div class="diff-side removed"><pre>- removed lines</pre></div>
    <div class="diff-side added"><pre>+ added lines</pre></div>
  </div>
  <div class="diff-annotation">Annotation on the important hunk</div>
</div>

<!-- Before/after schema card -->
<div class="before-after">
  <div><div class="ba-label">Before</div><!-- schema content --></div>
  <div><div class="ba-label">After</div><!-- schema content --></div>
</div>
```

## Output Instructions

1. Gather the actual diff before writing — never invent paths or changes.
2. Write the complete HTML file — never truncate.
3. Run `open <path>` immediately after writing.
4. State the file path, which tabs it contains, and how many files are in the key changes tab.
5. Do not commit the recap file unless asked.
